The ideal solution isn't always realistic

Research showed the documentation's IA didn't match how users approach tasks. The core IA issue remained due to legacy system constraints. The docs got prettier and more readable — but a fundamental restructure would require too much engineering effort.

Look for overlooked problems

No one was filing bug reports about documentation. The Control Panel worked well, which hid serious issues in the CLI and API docs. Research helped uncover a problem that wasn't visible in standard feedback channels.

Problem

Developers who knew exactly what they were doing
still felt like they were doing it wrong.

DigitalOcean's CLI/API documentation was technically complete. Everything was there — somewhere. But users kept hitting dead ends, questioning themselves, second-guessing steps. The information existed. The experience didn't.

Research Process

I needed to watch people fail —
not ask them how it went.

Who I recruited

Recruited via Recruitly with a deliberate 50/50 split: advanced developers and beginners, all in the software industry. If experienced developers were getting lost, the problem was structural — not a knowledge gap.

Why moderated think-aloud

Moderated think-aloud was the only method that surfaces the inner monologue — the moment confidence breaks, not a retrospective score. A survey would have told me how users felt. Watching told me exactly when and why.

Three failure patterns. One root cause.

Every participant struggled — but the friction clustered into three distinct patterns, each at a different layer of the system. All pointed to the same underlying issue: the documentation was organized around DigitalOcean's internal structure, not a developer's task flow.

Finding 01

They couldn't find it to begin with

Users looked in four places — search, left nav, most-viewed list, support tab. None reliably surfaced the CLI/API docs. The SSH section was its own maze: eight separate pages for one task.

"You don't even get curious to click on it."— Apostolos, Data Director
→ DecisionCLI/API added as first-class left nav item; 8 SSH pages flagged for consolidation.
Finding 02

The content left them stranded mid-task

No prerequisites. Undefined terms. And nobody was told a password would arrive by email after Droplet creation — every single participant assumed they'd made an error.

"I ended up assuming there was no password."— Karim, Software Developer
→ DecisionPrerequisites block and inline expectation-setting copy added to Droplet creation flow.
Finding 03

Navigation turned one task into four

Every redirect broke momentum. Some chains went three levels deep before arriving anywhere useful — pulling developers completely out of their mental context.

"It's taken my focus completely away from actually creating the droplet."— Ashwin, Cloud Architect
→ DecisionRegion slugs, image IDs, token flows recommended inline. Redirects to point to sections, not page roots.
